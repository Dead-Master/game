# 🎯 Обзор проекта

Это Laravel-приложение с браузерной пошаговой карточно-тактической игрой (hotseat на одном устройстве).  
Состояние игры хранится в SQLite и синхронизируется через API (AJAX).  
Основная игровая логика сосредоточена в `App\Services\GameManager`.

---

## 🛠 Технологический стек

- **Backend:** PHP 8.3, Laravel 13.17.0
- **База данных:** SQLite (`database/database.sqlite`)
- **Frontend:** Blade, JS (без SPA-фреймворка), Tailwind CSS v4, Vite 8
- **Тесты:** PHPUnit, Mockery
- **Пакетные менеджеры:** Composer, npm

---

## 📐 Архитектура и принципы

- Классическая MVC-структура Laravel.
- Контроллеры отвечают за HTTP-уровень (валидация, ответы).
- Игровая доменная логика должна жить в `GameManager` (или отдельных сервисах домена).
- Модели Eloquent используются напрямую, без лишних abstraction-слоев.
- Правила игры валидируются **на сервере**; клиентская проверка — только UX.

---

## 🗄 Ключевые сущности

### `games`
- состояние сессии игры;
- текущий ход/раунд;
- имена игроков;
- статус (`active/finished`).

### `game_players`
- сторона (`player_1` / `player_2`);
- здоровье и атака штаба;
- ресурсы;
- рука/колода (`hand`, `deck` в JSON).

### `units`
- юниты на поле;
- боевые статы;
- позиция;
- состояние (`board`, `graveyard`);
- флаги хода (`has_attacked_this_turn`, `has_counter_attacked_this_turn`).

---

## 🎮 Актуальные правила (важно)

1. Поле: `5x3`, координаты `x: 0..4`, `y: 0..2`.
2. Штабы:
    - `player_1`: `(0,0)`
    - `player_2`: `(4,2)`
3. Размещение карт — только в соседние клетки со своим штабом.
4. Перемещение в занятую клетку **запрещено**.
5. Атаки выполняются отдельными API-методами (не через move).
6. Контратака:
    - `infantry`, `scout` — могут;
    - `berserker` — если сам ещё не атаковал в этот ход;
    - `archer` — не контратакует.
7. У базы — отдельная атака 1 раз за ход.
8. Победа: `base_hp` противника <= 0.

## 🔌 API маршруты

- `POST /api/games` — создать игру
- `GET /api/games/{id}` — состояние игры
- `POST /api/games/{game}/deploy-card`
- `POST /api/games/{game}/move-unit`
- `POST /api/games/{game}/attack-unit`
- `POST /api/games/{game}/attack-base`
- `POST /api/games/{game}/end-turn`

## 🧪 Тестирование

- Основной файл API-тестов: `tests/Feature/GameApiTest.php`.
- При добавлении новой механики:
    1. Добавить позитивный сценарий;
    2. Добавить негативный сценарий;
    3. Проверить флаги/статусы в БД.

Запуск: bash 
vendor/bin/phpunit

---

## 💻 Frontend-договорённости

- Основной JS: `public/js/game.js`.
- Любые визуальные подсказки (курсоры, превью урона, анимации) не должны подменять серверную валидацию.
- Для дальних атак (лучник/штаб) используется лук-курсор и анимация стрелы.
- Для ближнего боя — короткая мили-анимация.

---

## 🔒 Безопасность и качество

- Не возвращать во внешние API “сырые” тексты исключений в production.
- Не полагаться на `side` только с клиента — желательно привязывать к пользователю при включении auth.
- Избегать дублирования правил игры между JS и PHP (истина — backend).

---

## ⚙️ Практика изменений

При изменениях:
1. Сначала правим доменную логику (`GameManager`), затем контроллер, затем UI.
2. Обновляем/добавляем тесты.
3. Проверяем совместимость с текущими правилами игры.
4. Не использовать PHP-фичи выше 8.3.
